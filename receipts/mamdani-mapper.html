The mapper is on raghavahuja.com/mamdani-mapper. This is the receipt — the architecture, the place reality pushed back, and the things I deliberately did not build.

the brief

Make something viral about Zohran Mamdani for a personal website. Turn open data into a map people want to screenshot. Don't be creepy. Don't pay for anything. Don't build infrastructure that needs babysitting.

The map is a receipt, not a tracker. Every event has a published source URL with a verifying quote. Even with stalker-bait flagged off, an aggregated real-time feed of an elected official's location enables the worst use case as cleanly as the best one.

three decisions

1. press receipts, not real-time tracking

Hard rule: the map only plots what the press office has already published. The methodology section on the page says this in plain English.

2. standalone page, not embedded in the site shell

The rest of raghavahuja.com is a portfolio site. The mapper is its own thing — a special bulletin from "the cartography desk" — with its own tokens, font stack, footer. Only handshake back is a ↩ raghavahuja.com link in the footer and a return entry in ⌘K.

3. cobalt face markers — not the obvious red pin

The default Mapbox marker is a red pushpin. Boring. The narrative hook is "Mamdani's face on the map," so the marker is his face, in a circle, with cobalt rings: solid for past, dashed for upcoming. The legend is honest: upcoming · web vs earlier · press.

architecture

Two scrapers in one GH Action. Both write to the same events.json.

scraper a · the press receipt (every 5 minutes)

  1. Discovery: nyc.gov mayor's-office sitemap is public XML.
  2. Skip-cache: URLs we've already extracted live in skip-cache.json. Most runs evaluate zero new URLs.
  3. Pre-filter: Executive Orders, "Statement on…" titles skipped before any LLM call.
  4. Extraction: Claude Haiku 4.5 with tool-use returns { was_physically_present, venue, neighborhood, borough, kind }.
  5. Geocoding: Mapbox geocoder with neighborhood + borough as hints — turned "La Marqueta" geocoding to a random street in Brooklyn into "La Marqueta, East Harlem, Manhattan."

scraper b · the forward signal (once a day)

One Claude API call/day with built-in web_search + web_fetch tools. Whitelisted NYC outlets. Strict criteria: specific venue, specific date inside 14-day window, public source URL, source explicitly states the mayor will attend. Returns a structured array via tool use.

the challenges

  • The press feed isn't a press feed. Mamdani's office, unlike Adams', doesn't publish a daily public schedule. Pivoted from "where will he be tomorrow" to "where has he been," with forward signal layered separately.
  • nyc.gov is JS-rendered. The index page is Adobe Experience Manager + Coveo. Static fetch sees an empty shell. Fix: the sitemap.
  • Geocoding the wrong "La Marqueta." "Gracie Mansion" pinned to Mansion Street in the Bronx. Fix: pass borough + neighborhood into geocoder query.
  • Rate limits at $5 of input tokens. 50K/min cap on Haiku, 30 of 60 calls hit 429s. Fix: 2.5s sleep, retry-after backoff, skip-cache.
  • "Today" view was empty by design. Press releases lag 1–3 days. Fix: rolling 7-day window default, plus an upcoming toggle.

what i deliberately didn't build

OFF-LIMITS

  • Crowd submissions. User-flagged: "fuck crowd submissions, people are creeps."
  • Twitter / X scraping. $5–200/mo for marginal-value signal. Skipped.
  • Real-time inference. No "he just left X" extrapolation. The map is a receipt.
  • The Politico email pipeline. Built a runbook for it. Decided to ship the web-search forward signal first.

cost engineering

The skip-cache is the most important single line of code. Without it: 288 runs/day × 50 URLs = 14,400 LLM evaluations/day. With it: 0–3 new URLs and exit. The cache is a JSON file in the same repo — committed alongside events.json on every change.
