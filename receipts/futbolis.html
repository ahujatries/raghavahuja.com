the problem

geographic disconnect

Modern football apps — FlashScore, FotMob, ESPN — present matches as data rows. Optimized for speed and density, but they strip away something fundamental: the sense that football is a global, living phenomenon. At any given moment there are 50+ professional matches happening across the planet. A user checking scores sees:

Slovakia 3. Liga - East
SPO 4 - 1 RAS

What they don't feel is that this match is happening in Spišské Podhradie, a town of 3,800 people at the foot of a medieval castle in eastern Slovakia.

What if scores weren't consumed — they were explored? What if checking live matches felt like surveying a living, breathing planet of football activity?

the solution — macro to micro

  • World view (macro) — entire globe with clustered match markers. South America glows during evening kickoffs, Europe during afternoon matches.
  • Region view (meso) — individual match markers, team codes (e.g., "STR vs OLI"), pulsing for live status.
  • City view (micro) — selecting a match triggers a fly-to that drops the user into the host city. Bold environmental typography overlays the city name ("Capiata, Paraguay" in 120px white text).

mobile: a complete rethink, not a shrunk-down desktop

Solution: a bottom-sheet carousel. Horizontal swipeable cards anchored to the bottom of the screen. Swiping a card into center focus triggers a map fly-to automatically. Tapping an active card expands it to a full detail view, hiding others to reduce cognitive load.

Building mobile as a separate architecture (a distinct MobileCarousel component, not a media-query on the sidebar) is the decision that made the phone experience feel native. It's more code. It was the right call.

the $19/month fix — intelligent edge caching

The football data API charges per request. A viral World Cup moment could mean 100,000 users hitting simultaneously — thousands of dollars in overage fees per hour. On a solo, ad-free project, that is bankruptcy.

// app/api/matches/route.ts
export const revalidate = 60

export async function GET() {
  const response = await fetch(
    'https://api-football-v1.p.rapidapi.com/v3/fixtures?live=all',
    {
      headers: {
        'X-RapidAPI-Key': process.env.RAPIDAPI_KEY!,
        'X-RapidAPI-Host': 'api-football-v1.p.rapidapi.com',
      },
      next: { revalidate: 60 },
    }
  )
  return Response.json(transformFixtures(await response.json()))
}

Every user hits the Vercel Edge cache, not the upstream API. At 100,000 concurrent users, the upstream API sees one request per minute. Cost scales with time, not traffic.

WHY NOT REDIS Most engineers would reach for a Redis cache or a Cloudflare Worker. I specifically chose Next.js's built-in revalidate because it's the minimum viable caching layer — no extra infrastructure, no new failure mode, no additional bill.

the 4-tier geocoding fallback

The API provides stadium names but not coordinates. "Estadio Erico Galeano" means nothing to Mapbox without lat/lng.

async function resolveMatchLocation(match) {
  // Tier 1: Stadium coords cache (~500 major venues)
  const stadium = STADIUM_DATABASE[match.venue.id]
  if (stadium) return { ...stadium, source: 'stadium', approximate: false }

  // Tier 2: City-level via Mapbox
  const city = await geocodeCity(match.venue.city, match.country)
  if (city) return { ...city, source: 'city', approximate: true }

  // Tier 3: Country centroid
  const country = COUNTRY_CENTROIDS[match.countryCode]
  if (country) return { ...country, source: 'country', approximate: true }

  // Tier 4: National team logic (international friendlies)
  if (isNationalTeam(match.homeTeam)) {
    return { ...CAPITAL_CITIES[match.homeTeam.country], source: 'national-team' }
  }
  return { ...FALLBACK_COORDS, source: 'unresolved', approximate: true }
}

Each match carries a locationApproximate boolean. The UI displays "Location is approximate / Ubicación aproximada" when true. Lying to the user about precision is worse than admitting imprecision.
