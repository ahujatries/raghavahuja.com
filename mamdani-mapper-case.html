The Mamdani Mapper is a one-night build: a standalone, cobalt-themed map at raghavahuja.com/mamdani-mapper that plots every public stop NYC Mayor Zohran Mamdani has made — and the credibly pre-announced ones — using only published data. Past stops come from the Mayor's Office press feed (1–3 days lag); upcoming stops come from a daily structured web search across credible NYC outlets. No tweets. No inferences. No third-party signals. No crowd submissions. The face on the map is the same image, recast as a marker.

This case study is the receipts: every architecture decision, every place reality pushed back, every thing I deliberately did not build. Total monthly cost ceiling is around $1.30/mo. Stack is vanilla HTML, two scripts, and one GitHub Action.

The brief

Make something viral about Zohran Mamdani for a personal website. Turn open data into a map people want to screenshot. Don't be creepy. Don't pay for anything. Don't build infrastructure that needs babysitting.

The promise to the user (and to the mayor's safety) had to do real work in two directions:

  • For the audience: a clean, share-worthy artifact — Mamdani's face on a map, dotted across NYC, with credible sourcing visible on every pin.
  • For the subject: a hard rule that the page never plots a real-time location, never aggregates third-party signals, and never accepts crowd submissions. The mapper is a receipt, not a tracker.

The decisions

Press receipts, not real-time tracking

The first thing I argued against. A "where is the mayor right now" page is technically possible from public signals — tweets, press appearances, photo-op spotting — but conflates "public information" with "responsible to publish." Even with stalker-bait flagged off, an aggregated real-time feed of an elected official's location enables the worst use case as cleanly as the best one. So I made it a hard rule: the map only plots what the press office has already published. Every event has a source_url with a verifying quote stored alongside the coordinates. The methodology section on the page says this in plain English.

Standalone page, not embedded in the site shell

The rest of raghavahuja.com is a portfolio site with a specific masthead, nav, and footer. The mapper is its own thing — a special bulletin from "the cartography desk" — and forcing it into the site chrome muddies both. So mamdani-mapper.html declares its own tokens, its own font stack, its own footer. It can borrow the cobalt palette without inheriting the masthead. The only handshake back to the parent site is a ↩ raghavahuja.com link in the footer and a return entry in ⌘K.

Cobalt face markers — not the obvious "red pin"

The default Mapbox marker is a red pushpin. Boring. The narrative hook of the project is "Mamdani's face on the map," so the marker is his face, in a circle, with cobalt rings that mean different things: solid for past, dashed for upcoming, pulsing green for live. Forward (web-sourced, predicted) events are visually distinct from past (press-sourced, confirmed) events — the dashed border carries the epistemic difference. The legend is honest: upcoming · web vs earlier · press.

Architecture

Two scrapers, both running in one GH Action workflow. Both write to the same events.json.

Scraper A · the press receipt (every 5 minutes)

  1. Discovery: the nyc.gov mayor's-office sitemap is public XML and contains every press-release URL in the form /news/YYYY/MM/<slug>. We pull it once per run.
  2. Skip-cache: URLs we've already extracted (or already ruled out) live in mamdani/skip-cache.json. Most runs evaluate zero new URLs and exit cleanly with $0 spent on Claude.
  3. Pre-filter: obvious non-event titles (Executive Orders, "Releases Plan," "Statement on…") get skipped before any LLM call.
  4. Extraction: what survives gets handed to Claude Haiku 4.5 with a tool-use schema. The model returns { was_physically_present, venue, neighborhood, borough, kind }. If was_physically_present === false, we skip and remember the URL.
  5. Geocoding: Mapbox's geocoder, with neighborhood and borough passed as hints. This is the fix that turned "La Marqueta" geocoding to a random street in Brooklyn into "La Marqueta, East Harlem, Manhattan." The hint matters enormously for ambiguous proper nouns.

Scraper B · the forward signal (once a day)

Mamdani's office, unlike his predecessors', does not publish a daily public schedule. Politico's NY Playbook newsletter (which historically would have) is fully blocked at the web layer — 403 on every direct request, Wayback's most recent snapshot is two months stale, no public RSS. Email pipeline is real but a multi-hour build with user-side setup.

So Scraper B uses Claude with the built-in web_search_20260209 + web_fetch_20260209 server-side tools. One API call per day. The system prompt asks for forward events from a whitelist of NYC outlets (Gothamist, NY1, NY Times Metro, City & State, RPA press releases, etc.) with strict criteria: specific venue, specific date inside a 14-day window, public source URL, source explicitly states the mayor will attend. The model returns a structured array via tool use.

An exhaustive one-time pass with Sonnet 4.6 found two high-confidence events for the next 14 days — a 9/11 Memorial wreath-laying with King Charles III on April 29, and a mainstage talk at the 2026 RPA Assembly on May 1. The daily Haiku run keeps watching.

Challenges

The press feed isn't a press feed

I designed Scraper A around an Adams-era convention: the Mayor's Office posts a daily "Mayor's Public Schedule" press release that lists where he'll be that day. Mamdani's office doesn't do this. They publish event-specific releases after events. The whole "where will he be tomorrow" framing was structurally impossible from this source. The pivot was painful but the right call: ship the receipt of where he's been, layer forward signal in separately.

nyc.gov is JS-rendered

The press-release index page (/mayors-office/news) is rendered client-side via Adobe Experience Manager + Coveo search. A static fetch() sees an empty shell. Individual press release pages are static, but you can't list them via the index. The fix was the sitemap — XML, public, and updated within minutes of new releases.

Geocoding the wrong "La Marqueta"

First clean Claude run produced 14 events with great extraction quality but ugly geocoding: "La Marqueta" pinned in Canarsie, Brooklyn. "Gracie Mansion" pinned to Mansion Street in the Bronx. "WNYC" pinned to a street in Brooklyn. The fix was passing the borough and neighborhood Claude already extracted into the geocoder query. Disambiguation goes from rough to precise the moment the model and the geocoder share context.

Rate limits at $5 of input tokens

The first end-to-end run burned through the 50K input-tokens-per-minute org cap on Haiku and 30 of 60 calls hit 429s. Three small fixes: a 2.5s sleep between calls, a 429-with-retry-after backoff that retries up to twice, and the skip-cache (so subsequent runs do far fewer calls). Steady-state is now under the cap by orders of magnitude.

The "today" view was empty by design

Earliest version defaulted the map to "today." But press releases lag 1–3 days, so "today" was almost always empty — the page rendered as broken. Fix: rolling 7-day window as the default for past events, and a new upcoming toggle (the now-default) that shows today + next 14 days. Honest empty-state copy explains the lag in plain English so first-time visitors don't think the page is dead.

What I deliberately did not build

  • Crowd submissions. User-flagged: "fuck crowd submissions, people are creeps." Removed from scope before a line of code.
  • Twitter / X scraping. $5–200/month for a marginal-value signal. Mamdani's @ tweets are policy, not schedule. The mayor's office tweets are post-hoc. Skipped.
  • Real-time inference. No "he just left X" extrapolation, no aggregation across signals, no clever guesses. The map is a receipt; if it can't cite a published source, it doesn't plot.
  • The Politico email pipeline. Built a runbook for it. Decided to ship the web-search forward signal first and revisit if traction warrants the additional infra.

The skip-cache is the most important single line of code in the project. Without it, 288 runs/day × 50 candidate URLs = 14,400 LLM evaluations per day. With it, the 30-second steady-state job evaluates 0–3 new URLs and exits. The cache is a JSON file in the same repo — committed alongside events.json on every change.

Pivots and trade-offs

  • Forward-looking → receipt: the original "where will he be" pitch became "where has he been," with a small upcoming column on the side. The narrative is stronger this way: the map IS the value, the share moment is the heatmap of his time.
  • Email pipeline → web search: 2–3 hour build with user setup vs. a one-API-call daily search. The web search runs in the existing GH Action; nothing new to maintain. Same source quality on the high end (RPA press releases, Gothamist), worse on the low end. Acceptable.
  • Opus → Haiku: on the user's first cost-anxiety prompt I downgraded the per-press-release extractor from Opus 4.7 to Haiku 4.5. 5x cheaper, accuracy still good for this task because the prompt is highly structured and the field of decisions is narrow.
  • Multi-call agentic loop → single tool call: the forward scraper does one Messages API call with web_search, web_fetch, and a custom record_forward_events tool. No round-trip looping from the client side. Claude does the agentic work server-side and returns one structured array.

What's live now

  • The standalone page at raghavahuja.com/mamdani-mapper with cobalt design, custom topbar, and its own favicon (the face).
  • 14 past events plotted across all 5 boroughs, plus 2 high-confidence upcoming events (King Charles wreath-laying, RPA Assembly).
  • The borough tracker — "who gets the most mamdani?" — with a face-stack visualization for the last 30 days.
  • Click-to-zoom: click a row or face → map zooms to the venue with the popup, click off → flies back to the overview.
  • Three share-card generators (today, this week, heatmap) that produce branded 1200×630 PNGs from the canvas.
  • A GH Action firing every 5 minutes for the press scraper and once a day at 14:00 UTC for the forward scraper.

What I'd do differently

  • Build the email pipeline. The exhaustive Sonnet sweep found 2 events for a 14-day window. Politico's NY Playbook would catch a few more — specifically the small town halls and community appearances that don't get an indie-press writeup. Cloudflare Email Routing → Worker → GitHub commit is ~3 hours of work plus a one-time email subscription. If the page gets meaningful traction, that's the next move.
  • Pad the system prompt above the cache floor. Haiku's prompt cache requires > 4096 tokens. My system prompt + tool schema clocks in around 3.5K, so caching never triggers. With more few-shot examples I could push above the floor and get ~90% off the per-call cost. Trivial savings at current volume; matters at 100x scale.
  • Forward events as a separate visual layer. Right now they live in the same events.json with a flag. A separate forward.json would let a future "preview" view diff what changed since yesterday — the kind of thing that turns a static map into a daily destination.

Conclusion

One night, vanilla static page, two cron jobs, $1.30/month. The building constraint that made it work was the negative one: don't build a tracker, build a receipt. That single line of editorial discipline is what kept the scope honest, the cost low, and the page shippable in one session. The face on the map does the rest.

Built for sport. One static page, one Mapbox tile, one hand-written scraper, one daily Claude run, and a GitHub Action that commits the receipt to the repo.
